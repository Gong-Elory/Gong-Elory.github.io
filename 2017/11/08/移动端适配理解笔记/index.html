移动端适配理解笔记

发表于 2017-11-08   |   分类于 前端   |     |   阅读次数

自从发现了手淘的优雅适配移动端方案flexible之后一直有个疑问就是rem到底起着什么作用，因为如果设置了固定px样式后，发现在所有终端下都是相等的px，那到底弄这么复杂的移动端适配是为了干啥，今天看了几篇文章，貌似恍然大悟，免得以后忘记，写在这里。

第一：移动端开发必备的meta

这句话应该是信手拈来的，但表达的意思以前我都没搞太懂，我现在是这样理解的
width=device-width 这个设置是设置布局适口为理想视口，理想视口也就是这部手机的物理宽高。

比如ihpone6（之后的例子都是以6来说的）的宽375pt，所以这里的width为375px。布局适口我们就当作是程序员操作的界面。
如果我们后面设置的scale为1，也就是缩放比例为1，其实就是让我们的布局视口 = 1倍理想视口。

手淘的方案是这样搞的。根据设备宽高，也就是clientWidth，使用js动态设置scale。假设为375，我们设置scale为0.5。什么意思？

意思就是把我们的布局视口缩小0.5倍，为嘛要缩小。先得知道iphone6的物理像素，宽为750pt，代表一排有750个硬件像素点。我们布局视口宽375px，也就是说程序员操作的1px*1px像素点实际上是等于2*2 = 4个硬件像素点的.
缩小0.5倍实际上就是让程序员操作的1px * 1px = 硬件上的1个像素点 这样有啥好处呢。好处就是在我们可以直接用设计稿上多大的尺寸，我们写出的程序看起来就是多大的尺寸了。如果按照原来的1来看，从视觉上写出来的尺寸会比设计稿上尺寸大。
但是缩放0.5只是为了适应dpr为2的屏幕，对于其他分辨率的就不适用了。所以这里我们动态设置他，针对3的屏幕，他就是0.3333333.

第二： rem
但是这并没有解决问题，因为我们的设计稿是640px的。一个红方块是200px。我们操作的像素是750px，显示的也是200px的红方块。这样最终的布局肯定就会乱，所以到这里还没有结束。
这里就用到了rem，来根据不同的屏幕调整方块的大小了。
rem是相对根元素html的fontSize样式来计算的，所以我们要动态的设置这个值。这里手淘和网易采用两种不同的方法

  • 网易：
    根据设计稿的宽度计算一个基准值例如 640 / 100 = 6.4 这个6.4作为基准值。640哪里来，640用到真机上就是clientWidth 。所以iphon6就是375 那fontSize值到底怎样算呢。就用设计稿的基准值来算。
    fontSize = clientWidth / 6.4 那么设计稿中fontSize为100px 也就是1rem = 100px，这样写程序的时候就很方便。200px的方块就是2rem。 那这个盒子在375的clientWidth下多大呢。
    我们知道375下的fontSize为375 / 6.4 = 58.59px 我们写200px的盒子用的是2rem， 那么这里最终得出来的具体值就是 2 * 58.59px = 117px
  • 淘宝
    淘宝的fontSize = clientWidth dpr / 10。前面的clientWidth dpr是什么意思呢 算出来的其实就是物理像素大小 例如6的就是375 2 = 750，fontSize = 75px .前面我们说到为了让1个物理像素点等于一个程序员可以操作的点（CSS像素点）。我们设置了scale。现在这里就用上了。
    我们在设计稿中是640pxize，假设设计稿是1 那么fontSize = 640     1 / 10 = 64px ，然后我们就利用这个来算盒子的大小，为3.125rem 那么在6真机上出来的就是234px.虽说和网易计算出来的值不一样，但是在一部机器上绝对一样大小。原因就在scale身上。

那么网页中的字体，网易的基本上16px就直接这样写，因为scale没有变化。淘宝的可就不能一直16px了，因为已经让一个物理像素等于一个CSS像素了,在16px大小的字块从体积上宽高已经在6上缩小了4倍，所以要使用媒体查询更改字体大小。2倍屏为216
